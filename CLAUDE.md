# Fontainebleau fire &amp; spots ŌĆö working notes

Context for anyone (human or agent) working in this repo. `README.md` covers
what the map is, how to run it and how to deploy it; this file covers how the
project works internally, and the things that will waste your time if you
don't know them.

## What this is

A static Leaflet map showing the July 2026 Fontainebleau forest fire against
the parkour/bouldering spots in the massif, used to plan trips before
travelling. Plain HTML and JavaScript ŌĆö no build step, no bundler, no
framework, no API keys, no accounts. `index.html` is the whole application.

Data lives in `data/`. The `*.geojson` files are generated by the scripts in
`scripts/` ŌĆö dependency-free standard-library Python, safe to re-run at any
time, and they overwrite their output. `data/closures.pdf` is *not* generated:
it is a source document that cannot be re-downloaded by any script here, so
don't clear the directory to rebuild it.

## The fire is shown from two sources, deliberately

Neither source is sufficient alone, so both are on the map at once:

- **Copernicus EMS EMSR894** is the authoritative mapping of this fire and is
  current and precise, but it is only available as pre-rendered raster tiles.
  It is the primary visual layer, loaded live from their tile service, so it
  needs no refresh ŌĆö they update it in place.

  The tile URL is not documented on their site. It comes from their public
  StoryMap for the activation: fetch
  `https://www.arcgis.com/sharing/rest/content/items/<storymap-id>/data?f=json`
  and read the `url` of the web-map layers it references (`EMSR894_latest` is
  the self-updating one). Use the same route to wire up a future activation.
  The StoryMap for this fire is
  `https://storymaps.arcgis.com/stories/76b8c43a0cc04b36b7d0ac1efd0fd36b/`.
- **EFFIS** serves real polygons, but they are MODIS-derived: coarse
  (~250ŌĆō500 m) and updated with a lag of days. They provide the dashed outline
  and the geometry used to flag spots inside the burn.

**The EFFIS polygons understate the burnt area** (~1 581 ha against a reported
2 000+ ha). This matters: people use this map to decide where to go. A spot
that is not flagged is *not* known to be unburnt, and the legend says so. Keep
that caveat visible in any change you make to the fire layers or the legend.

## Things that do not work ŌĆö don't spend time on them

- **There is no vector version of Copernicus EMSR894.** Their ArcGIS
  organisation (`rNFYfizUMWjwfBKB`) hosts no feature services; the tile service
  reports `capabilities: Map,TilesOnly,Tilemap`, so `/query` returns 404; the
  portal has no API under any `/api/...` path; and the legacy
  `emergency.copernicus.eu/mapping/list-of-components/...` URL is dead. The
  raster tiles are the only usable product.
- **EFFIS `effis.nrt.ba.poly` errors server-side** (`msPostGISLayerGetItems`
  SQL error), even though it is the layer you would want. Use
  `modis.ba.poly.season`.
- **EFFIS GetCapabilities takes minutes.** Go straight to GetFeature.
- **OpenStreetMap only has one of the three closed forests**
  (`For├¬t Domaniale des Trois Pignons`, relation 16615234). Fontainebleau and
  la Commanderie do not exist there as forest boundaries, which is why the
  closed-area layer is not built. IGN's public forest dataset is the likely
  source if it is picked up later.

## Gotchas in the data

- **EFFIS returns coordinates latitude-first**, while GeoJSON requires
  longitude-first. `scripts/build_fire.py` detects and corrects this, and
  raises if a feature ends up outside the area of interest. Without the swap
  the burnt areas render off the coast of Somalia. Any new WFS source needs the
  same check ŌĆö both values are plausible numbers, so the only way to tell is
  the area of interest.
- **The Google My Maps KML encodes category in the `styleUrl` icon id**, not in
  the placemark name: `1680` = spot, `1644` = parking, `1578`/`1684` = shop,
  `1765` = campsite. Trust the icon id; names are inconsistent and some are
  truncated in the source map.
- **The KML contains ten city folders**, only one of which is used. The source
  map covers B├Čblingen, Sindelfingen, Herrenberg, Uni Vaihingen, Stuttgart,
  Ludwigsburg, M├╝nchen, Fontainebleau, Lisses/Evry and Berlin;
  `build_spots.py` selects the `Fontainebleau` folder by name. The other
  folders are a different project and must not leak onto this map.
- The KML has **no descriptions**, only names and coordinates, so popups can
  only show the name and the derived category.
- Some spot names are informal German nicknames from the original map. They are
  kept verbatim rather than corrected ŌĆö it is someone else's data.

## Verifying changes

There is no test suite, and the things that break here are visual. Serve the
directory and actually look at the result:

```sh
python -m http.server 8000
chrome.exe --headless=new --disable-gpu --hide-scrollbars \
  --window-size=1280,900 --virtual-time-budget=25000 \
  --screenshot=out.png http://127.0.0.1:8000/
```

Check **both** `1280,900` and `390,780` ŌĆö phones are a priority, and the two
layouts fail differently. On Windows, headless Chrome clamps the window to
~500 px wide: ask for `390,780` and the page lays out at ~500 px while the
screenshot is cropped to 390, so anything in the right-hand 110 px looks cut
off when it is not. Check `window.innerWidth` (`--dump-dom` on a page that
writes it into the title) before believing a narrow-layout bug, and use
`500,780` for a narrow check that isn't lying to you. Looking at the rendered image is what catches swapped
coordinates, legend colours that don't match the tiles, and controls that cover
the map. A syntax check alone proves nothing.

The page fetches GeoJSON, so it needs a real server; `file://` will not work.

## Conventions and constraints

- Keep it dependency-free and static. It is hosted on GitHub Pages from the
  repository root.
- Every layer keeps its own GeoJSON file so a new source is a contained change.
  There is intentionally **no** config/manifest system ŌĆö new data is wired in
  by hand.
- Base maps must stay free and account-free. IGN's G├®oplateforme
  (`data.geopf.fr`) serves Plan IGN and orthophotos with no key. IGN *Scan25*
  is **not** usable ŌĆö it needs a licence ŌĆö despite being the style used on the
  ONF closure map.
- Attribution for IGN, OpenStreetMap, Copernicus EMS and EFFIS is required and
  must stay on the map.
- The page is `noindex`, and the spots come from someone else's shared map, so
  don't publicise the URL.
- `data/closures.pdf` is an official ONF map, dated 17 July 2026, showing three
  state forests closed in full: **Fontainebleau**, **Trois-Pignons** and
  **la Commanderie**. It is the reference for *which* forests are closed. It is
  a scanned image with no text layer, so nothing can be extracted from it
  programmatically; it is not a geometry source.

## Deliberate omissions

Closed-area polygons, fire progression over time, damage-severity grading, and
geolocation. Geolocation is intentional: this is a planning map used before
travelling, not a navigation aid for use in the forest.
